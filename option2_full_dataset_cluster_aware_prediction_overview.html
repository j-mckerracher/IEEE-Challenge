Option 2: Full FRESCO Cluster-Aware Failure Prediction Challenge

A multi-cluster challenge concept using the full FRESCO dataset.

Overview

Welcome to the Full FRESCO Cluster-Aware Failure Prediction Challenge. In this competition, participants work with resource-usage data from multiple HPC systems represented in the FRESCO dataset. The data contains job metadata and resource telemetry such as CPU usage, memory usage, GPU activity, NFS traffic, and block-device I/O.

Your objective is to build a model that predicts whether a job will later fail or time out using job metadata, early resource-usage telemetry, and cluster/environment context.

In short: use the full FRESCO dataset to build a cluster-aware early-warning model for job failure.

Description

FRESCO contains data from multiple production HPC systems collected across different hardware generations, scheduler environments, and monitoring stacks. This option asks participants to build a single predictive pipeline that uses the entire dataset while respecting cluster-specific differences.

Participants will receive a training set containing jobs from all included clusters and a test set containing later jobs from those same clusters. Each row represents a job-level prediction instance constructed from job metadata and early telemetry.

The Problem

The dataset captures vital resource usage metrics across jobs, hosts, and clusters. The task is to predict a binary label:

  • 0: job completed successfully
  • 1: job failed or timed out

The challenge lies in handling heterogeneous environments. A pattern that is predictive on one cluster may not mean the same thing on another cluster, so participants must build models that are cluster-aware, robust, and carefully validated.

Important Note: This Is Not Zero-Shot Transfer

Recent FRESCO research suggests that zero-shot transfer to an unseen target cluster is not currently reliable. Therefore, this challenge should not be framed as “train on one cluster and predict another unseen cluster.”

Instead, this is a pooled multi-cluster prediction challenge: all target clusters are represented in training, and the goal is to model them jointly while accounting for cluster differences.

Metrics Captured

The source data is stored in monthly zip archives containing hourly Parquet files. The observed schema includes:

  • time: timestamp for the telemetry record
  • submit_time, start_time, end_time: job timing fields
  • timelimit: requested walltime limit
  • nhosts, ncores: requested hosts and cores
  • account, queue, host, jid, jobname, username: anonymized job/accounting fields
  • exitcode: final job state used to construct the label
  • host_list: allocated hosts
  • value_cpuuser: CPU user-mode activity
  • value_gpu: GPU active time percentage, when applicable
  • value_memused: physical memory usage
  • value_memused_minus_diskcache: memory usage excluding disk cache, when available
  • value_nfs: NFS data-transfer rate
  • value_block: block-device I/O rate

Cluster Context

Participants may receive cluster context such as:

  • cluster identifier
  • queue / partition
  • hardware regime, if available
  • time period / era features
  • documented measurement semantics, when available

This is important because a shared schema does not guarantee identical measurement meaning across clusters.

Allowed Modeling Techniques

Participants may use any supervised binary-classification method, including:

  • global pooled models
  • cluster-specific models with an ensemble or routing rule
  • hierarchical / mixed modeling ideas
  • gradient boosting or random forests
  • calibrated classifiers
  • neural models, if practical and well documented

Evaluation Criteria

Submissions will be evaluated based on predicted failure probabilities and actual job outcomes.

  • Prediction Accuracy (70%): ranked primarily by macro-average AUPRC across clusters. This prevents one large cluster from dominating the score.
  • Methodology and Creativity (20%): quality of cluster-aware modeling, leakage handling, treatment of heterogeneous measurement semantics, and interpretation of resource signals.
  • Efficiency and Documentation (10%): clean, modular, reproducible, and computationally efficient code.

Secondary diagnostics may include overall AUPRC, AUROC, per-cluster AUPRC, calibration, and recall at fixed false-positive rate.

Why Not R or R²?

This challenge predicts whether a job fails or times out, so it is a classification task. Correlation or R² would be more appropriate for regression tasks such as predicting memory usage or runtime. For rare-event failure prediction, AUPRC is a better primary metric.

Use of LLMs

Use of LLMs or AI tools should be disclosed. Participants should explain how AI tools were used and how their outputs were validated.

CSV Submission

The submitted CSV should contain one row for each row in the test set and must include:

  • row_id: identifier from the test data
  • failure_probability: numeric probability between 0 and 1

Higher probabilities should indicate greater predicted risk of failure or timeout.

Expected Deliverables

  • Prediction CSV: one probability per test row
  • Python code or notebook: feature construction and model training
  • Short report: methods used, validation results, cluster-wise performance, leakage controls, and interpretation of important features
  • Optional video: brief explanation of the approach

Dataset Description

The source FRESCO data can be downloaded from https://www.frescodata.xyz/bulk-download. Monthly files are provided as {YYYY-MM}.zip archives containing hourly Parquet files.

For this challenge option, the organizer would derive participant-ready CSV files from all included clusters, likely including:

  • train.csv: labeled jobs from earlier time periods across all included clusters
  • validation.csv: labeled jobs from later periods across all included clusters
  • test.csv: unlabeled jobs from newest periods across all included clusters
  • sample_submission.csv: required submission format
  • cluster_metadata.html: cluster context and known measurement differences
  • data_dictionary.html: feature definitions and leakage rules

Recommended Split Design

  • Use temporal splits within each cluster
  • Pool cluster-specific training windows into one training set
  • Use later windows from each represented cluster for validation and hidden test
  • Report both overall and per-cluster performance

Bottom Line

This option uses the full dataset and is more ambitious than the Anvil-only version. It should be framed as pooled, cluster-aware prediction across known clusters — not as zero-shot transfer to unseen clusters.
