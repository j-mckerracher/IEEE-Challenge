Option 1: Anvil Early Job Failure Prediction Challenge

A single-cluster challenge concept modeled after the Kaggle-style overview/data tabs.

Overview

Welcome to the Anvil Early Job Failure Prediction Challenge. In this competition, participants work with resource-usage data from the Anvil supercomputer. The data contains job metadata and time-series resource telemetry such as CPU usage, memory usage, GPU activity, NFS traffic, and block-device I/O.

Your objective is to build an early-warning model that predicts whether a job will later fail or time out using only information available early in that job’s execution.

In short: given early telemetry from an Anvil job, predict whether that job will complete successfully or end in failure/timeout.

Description

Data centers and supercomputing centers run thousands of jobs from many users. Some jobs fail, time out, or terminate abnormally after consuming substantial system resources. If operators can identify risky jobs early, they can alert users, adjust policies, or reduce wasted compute time.

Participants will receive a training set of Anvil jobs with known outcomes and a test set of Anvil jobs where outcomes are hidden. Each row represents a job-level prediction instance constructed from job metadata and early resource-usage telemetry.

The Problem

The dataset captures job activity across hosts and time. For this challenge, the raw telemetry will be transformed into early-window features. The goal is to predict a binary label:

  • 0: job completed successfully
  • 1: job failed or timed out

The challenge lies in using early resource patterns to distinguish jobs that are likely to fail from jobs that are likely to complete normally.

Why Anvil Only?

Recent FRESCO research suggests that cross-cluster model transfer is unreliable when clusters use different hardware, workloads, and measurement semantics. Therefore, this option focuses on a single, dense, recent cluster: Anvil. This keeps the challenge focused on operational prediction rather than cross-cluster transfer artifacts.

Metrics Captured

The source data is stored in monthly Parquet files and contains columns such as:

  • time: timestamp for the telemetry record
  • submit_time, start_time, end_time: job timing fields
  • timelimit: requested walltime limit
  • nhosts, ncores: requested hosts and cores
  • account, queue, host, jid, jobname, username: anonymized job/accounting fields
  • exitcode: final job state used to construct the label
  • host_list: allocated hosts
  • value_cpuuser: CPU user-mode activity
  • value_gpu: GPU active time percentage, when applicable
  • value_memused: physical memory usage
  • value_memused_minus_diskcache: memory usage excluding disk cache, when available
  • value_nfs: NFS data-transfer rate
  • value_block: block-device I/O rate

Allowed Modeling Techniques

Participants may use any supervised learning method suitable for binary classification, including:

  • logistic regression
  • random forests or gradient boosting
  • calibrated classifiers
  • time-series feature engineering from early telemetry
  • neural models, if computationally practical and well documented

Participants should be careful to avoid leakage: features derived from the end of the job, the final runtime, or the exit state must not be used as predictors.

Evaluation Criteria

Submissions will be evaluated based on how well predicted failure probabilities match actual job outcomes.

  • Prediction Accuracy (70%): ranked primarily by AUPRC on the hidden test set. AUPRC rewards finding failed jobs without raising too many false alarms.
  • Methodology and Insight (20%): quality of modeling approach, leakage handling, feature reasoning, and interpretation of resource-usage signals.
  • Efficiency and Documentation (10%): clean, modular, reproducible, and computationally efficient code.

AUROC, calibration, and recall at fixed false-positive rate may be reported as secondary diagnostics.

Use of LLMs

Use of LLMs or AI tools should be disclosed. If participants use AI tools for coding, explanation, or modeling ideas, they should describe how the tools were used and what human validation was performed.

CSV Submission

The submitted CSV should contain one row for each row in the test set and must include:

  • row_id: identifier from the test data
  • failure_probability: numeric probability between 0 and 1

Higher probabilities should indicate greater predicted risk of failure or timeout.

Expected Deliverables

  • Prediction CSV: one probability per test row
  • Python code or notebook: feature construction and model training
  • Short report: methods used, validation results, leakage controls, and interpretation of important resource signals
  • Optional video: brief explanation of the approach

Dataset Description

The source FRESCO data can be downloaded from https://www.frescodata.xyz/bulk-download. Monthly files are provided as {YYYY-MM}.zip archives containing hourly Parquet files.

For this challenge option, the organizer would derive participant-ready CSV files from the Anvil subset, likely including:

  • train.csv: labeled Anvil jobs from an earlier time period
  • validation.csv: labeled Anvil jobs from a later time period
  • test.csv: unlabeled Anvil jobs for final scoring
  • sample_submission.csv: required submission format
  • data_dictionary.html: feature definitions and leakage rules

Bottom Line

This is the cleanest and safest challenge option: use Anvil only, predict job failure/timeout from early telemetry, and grade with AUPRC plus a short methodology report.
