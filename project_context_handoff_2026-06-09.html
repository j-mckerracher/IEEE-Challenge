Core IEEE Challenge Direction

The first and currently cleanest challenge option is an Anvil-only early job failure prediction challenge.

Given early job metadata and resource-usage telemetry from Anvil, predict which jobs are likely to fail or time out before they finish.

Reasoning:

  • The prompt is about analyzing resource usage and predicting events such as failures or performance slowdowns.
  • Failure/timeout is clearer and easier to score than slowdown.
  • Anvil was selected because the user preferred the cluster with high usable data density and volume.
  • A single-cluster challenge avoids turning the competition into a cross-cluster transfer problem.

Option 1: Anvil-Only Failure Prediction

  • Dataset scope: Anvil only
  • Task: binary classification for failure/timeout
  • Inputs: job metadata + early telemetry features
  • Recommended feature packaging: precomputed early-window features
  • Primary metric: AUPRC
  • Secondary metrics: AUROC, calibration, recall at fixed false-positive rate
  • Split: temporal train/validation/test

Relevant file: option1_anvil_failure_prediction_overview.html

Option 2: Full-Dataset Cluster-Aware Prediction

A second option was generated using the entire FRESCO dataset.

Use the full FRESCO dataset to build a pooled, cluster-aware early-warning model for job failure.

Important nuance:

  • This option is not zero-shot cross-cluster transfer.
  • It is a pooled multi-cluster prediction challenge where the target clusters are represented in training.
  • Because one cluster could dominate the leaderboard, the recommended primary metric is macro-average AUPRC across clusters.

Relevant file: option2_full_dataset_cluster_aware_prediction_overview.html

Why AUPRC Instead of R or R²?

Failure prediction is a classification task, not a regression task. Therefore, correlation, R, or R² are not the right main metrics unless the task is changed to predicting a continuous value such as memory usage, runtime, or wait time.

  • Failure prediction: use AUPRC / AUROC
  • Resource amount prediction: use R² / MAE / RMSE

FRESCO Research Summary

The relevant paper is titled:

Beyond Unified Schemas: Why Zero-Shot Cross-Cluster Workload Transfer Fails and What It Takes to Make HPC Comparisons Defensible

Main findings discussed:

  • Zero-shot cross-cluster memory prediction failed badly.
  • Failure persisted after hardware/regime matching, overlap-aware cohort selection, feature simplification, dense sampling, CORAL, quantile mapping, and few-shot calibration.
  • Few-shot calibration with up to 500 labeled target examples did not rescue transfer.
  • Local Anvil modeling produced meaningful positive results, suggesting local site-specific modeling is currently more defensible.

Important Research Nuance: Pooled Models vs Transfer

Earlier FRESCO work did try pooled models such as pooled_no_id and pooled_with_id. Those models performed reasonably when the target cluster was represented in training. However, this is different from zero-shot transfer to a target cluster absent from training.

The refined claim is therefore: pooled multi-cluster supervised modeling may work when all target domains are represented, but zero-shot or near-zero-shot deployment to an unseen target cluster remains unreliable.

Why Transfer Is Not Currently Feasible

The main obstacle is not only distribution shift. It is also measurement-semantic non-equivalence.

Example: two clusters may both report “memory,” but one may include page cache while another excludes page cache or reports RSS. The shared column name does not imply the same operational quantity.

  • Measurement non-equivalence: different meaning of the target/feature
  • Covariate shift: different hardware/workloads/schedulers
  • Conditional shift: same inputs map to different observed outcomes
  • Era shift: different time periods, software stacks, and user communities

Meeting Direction With Saurabh

The desired meeting framing is strategic rather than nitty-gritty. The preferred sequence is:

  1. Publish the argument that provenance and measurement-semantic standards are necessary.
  2. Then study cross-cluster comparability as an open research question.
  3. Only later consider rebuilding or redesigning the dataset around comparability goals.

Key framing:

Schema harmonization alone is insufficient; provenance standardization is a necessary precondition for defensible cross-cluster analysis.

Current Bottom Line

  • Option 1 is cleaner and safer: Anvil-only early failure prediction.
  • Option 2 is broader and uses the whole dataset: pooled cluster-aware prediction, not zero-shot transfer.
  • The research story should emphasize the need for provenance standards before advocating major community data collection changes.
