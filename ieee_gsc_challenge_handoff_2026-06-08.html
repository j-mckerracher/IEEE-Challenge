IEEE GSC Challenge Handoff

Date: 2026-06-08

Purpose

This handoff is meant to start a new conversation focused on IEEE Computer Society Global Student Challenge 2026 challenge design.

Prompt Context

Analyzing Resource Usage in Data Center Computers and Predicting Events (i.e. Failures or Performance Slowdowns)

Current Recommendation

Early HPC job failure prediction on a single cluster (Anvil) using job metadata and early resource-usage telemetry.

Key Decisions Reached So Far

1. Prediction, not telemetry cleaning

Telemetry cleaning/provenance is relevant, but it should not be the main challenge. The prompt is more directly about using resource data to predict operational events.

2. One cluster, not many

Use one cluster only for the main leaderboard task. Recent research indicates cross-cluster transfer is not reliable enough to be the centerpiece.

3. Selected cluster: Anvil

Anvil was chosen under the principle: use the cluster with the highest density and greatest volume of usable data.

4. Early-warning framing

Teams should only get information available early in the run. The task should not become postmortem classification using full-job information.

5. Failure/timeout as the primary target

Recommended:

  • primary label: failure/timeout
  • secondary / bonus: slowdown, if included at all

Working Challenge Summary

Build a model that predicts whether an HPC job on Anvil will fail or time out, using only job metadata and early resource-usage telemetry.

Recommended Inputs

  • job/request metadata
  • early CPU / memory / I/O / related telemetry summaries
  • possibly queue/partition and timing features
  • ideally as precomputed early-window features rather than raw time series

Evaluation Direction

The dataset itself provides the ground truth through actual job outcomes. The organizer does not need a separate grading model.

Recommended leaderboard setup:

  • teams submit risk scores / probabilities
  • compare predictions against real failure outcomes

Recommended metrics:

  • Primary: AUPRC
  • Secondary: AUROC

Evaluation Design Principles

  • use temporal splits, not random splits
  • make the prediction-time feature boundary explicit
  • exclude full-runtime or post-event fields
  • avoid leakage and direct label proxies

Slowdown Discussion Status

Slowdown remains possible, but should probably be secondary / bonus only, not the main leaderboard target.

Research Basis Behind These Decisions

These decisions were influenced by the MASCOTS 2026 paper:

Beyond Unified Schemas: Why Zero-Shot Cross-Cluster Workload Transfer Fails and What It Takes to Make HPC Comparisons Defensible

Relevant takeaway: cross-cluster predictive transfer failed badly, so the safest challenge design is local, single-cluster, and prediction-time realistic.
